4WD Tyre Pressure Monitor & Auto-Inflator — Raspberry Pi 5
============================================================

SYSTEM OVERVIEW
============================================================
- 1x SSD1309 OLED (I2C): 4WD outline with live pressure at each wheel and
  a preset list (ROAD/DIRT/SAND/CUSTOM) down the right side. Flashes
  full-screen warnings for punctures / over-pressure / stuck-deflate /
  possible compressor failure (see below).
  (There is no on-screen speed/stage header — the 128x64 panel is fully
  occupied by the 4WD diagram, the F/R target labels and the preset list.
  GPS speed and lockout stage drive behaviour internally but are not drawn.)
- 1x DFRobot Gravity ADS1115 4-channel ADC reading pressure sensors
  (0.5-4.5V = 0-150psi), one sensor per tyre.
- Air solenoids (inflate + deflate per corner) via relay/MOSFET boards.
- 1x MOSFET power-switch module turning the air compressor on/off (it
  has its own pressure-switch cutout, so the Pi just powers it on/off).
- 3x momentary push buttons: UP, DOWN, SELECT (preset cycle / axle focus
  / silence warnings — see WARNINGS section).
- 1x u-blox NEO-6M GPS module (UART/NMEA) for speed-based lockout.

============================================================
SPEED-BASED LOCKOUT (2-stage)
============================================================
    Stage 0  (< 20 km/h)   : full access — all presets, all buttons.
    Stage 1  (20-80 km/h)  : only ROAD/DIRT/SAND selectable (CUSTOM
                              skipped). UP/DOWN can still fine-tune
                              whichever is active.
    Stage 2  (> 80 km/h)   : buttons fully locked out — UP/DOWN/SELECT
                              do nothing (except silencing a warning —
                              see below).

  IMPORTANT: the lockout ONLY ever gates the buttons/UI. Sensor
  monitoring and the automatic inflate/deflate control loop keep
  running at every speed, including above 80 km/h — a tyre that's off
  target keeps getting corrected regardless of lockout stage. "Locked
  out" means the driver can't change settings while moving, not that
  the system stops maintaining pressure.

  FAIL-SAFE: no valid GPS fix, or no data for GPS_FIX_TIMEOUT_SECONDS,
  -> speed treated as UNKNOWN -> falls back to Stage 2 (most
  restrictive button lockout), never assumed to be 0 km/h.

============================================================
WARNINGS: puncture vs. possible compressor failure
============================================================
Previously a corner that couldn't reach its target within
WARNING_TRIGGER_SECONDS just gave up and parked itself. That's been
replaced with:

  - The corner KEEPS TRYING indefinitely — it never stops attempting to
    inflate a tyre that's losing pressure. There's no permanent "fault
    lock" for this case anymore (sensor read errors / out-of-range
    sensor voltages are still a hard fault, since there's no trustworthy
    pressure data to safely act on at all).

  - After WARNING_TRIGGER_SECONDS of continuously trying to inflate
    without reaching target, the system tries to work out why:
        * If a tyre on the OTHER axle is inflating/holding fine ->
          this is a LOCAL problem -> "<POSITION> TYRE PUNCTURE".
        * If no other-axle tyre can be confirmed working either ->
          "POSSIBLE COMPRESSOR FAILURE" (system-wide, not tied to one
          wheel).
      (This is a simple heuristic based on whether other corners
      currently have their own active warning, not an active
      diagnostic test of the other axle — a reasonable first pass
      given the hardware available.)
      NOTE: with only one corner wired up for testing right now, there
      is no "other axle" to check, so any prolonged inflate struggle on
      the test corner will always resolve to "POSSIBLE COMPRESSOR
      FAILURE" until a second corner (on the other axle) is wired in.

  - Whichever warning fires, the OLED flashes it full-screen: white
    background, black text, 2 seconds on / 2 seconds back to the
    normal dashboard, looping for as long as the condition is active.
    The flashing screen shows "Push any button to pause warning" in
    small text at the bottom.

  - Pressing ANY button while a warning is flashing silences the
    flashing (that button press does nothing else — it's consumed by
    the warning) and returns to the normal dashboard, but the affected
    tyre's pressure reading is now shown in an inverted box (white box,
    black text) as a persistent reminder. A possible-compressor-failure
    warning, not being tied to one wheel, inverts the front/rear target
    labels instead. This silencing works even during Stage-2 speed
    lockout, since you should be able to silence a distracting warning at
    any speed.

  - The highlight/silence clears automatically once the underlying
    problem actually resolves (pressure recovers, or another corner
    proves air is flowing again).

============================================================
SENSOR FAULT DETECTION
============================================================
Each pressure sensor outputs 0.5-4.5V across 0-150psi. A reading OUTSIDE
that band (below SENSOR_V_FAULT_LOW or above SENSOR_V_FAULT_HIGH) means
the sensor is disconnected, shorted, or otherwise untrustworthy — NOT a
genuine 0psi reading. Those are treated as a hard fault for that corner:
solenoids close, the corner shows "ERR", and it does NOT inflate. This
prevents the (now never-give-up) inflate loop from inflating blindly
toward a wrong "0psi" reading from a dead sensor. The fault is
self-healing — it clears automatically as soon as the sensor returns a
valid in-band reading again. Each cycle also takes several samples and
uses the median, to reject electrical noise around the tolerance band.

============================================================
OVER-PRESSURE GUARD & STUCK-DEFLATE WARNING
============================================================
- Independently of the selected preset target, any corner reading above
  HARD_MAX_PSI is treated as a runaway and force-deflated (compressor stays
  off, deflate solenoid opens) with an "<POSITION> TYRE OVER-PRESSURE"
  warning, until it drops back into range. HARD_MAX_PSI sits comfortably
  above MAX_PSI so normal regulation never trips it.
- The deflate path now has its own struggle warning, mirroring the inflate
  one: a corner that can't reach target by DEFLATING within
  WARNING_TRIGGER_SECONDS raises "<POSITION> TYRE WON'T DEFLATE" (a jammed
  vent valve is always a local problem, so it's never attributed to the
  compressor). It keeps trying regardless.

============================================================
COMPRESSOR + AIR TANK (tank-pressure controlled)
============================================================
The compressor and the corner solenoids are DECOUPLED. The compressor's
only job is to keep an air TANK charged; the corner solenoids draw from
that tank on demand, independently.

A dedicated tank pressure sensor drives the compressor with hysteresis
(TankController): the compressor turns ON at/below TANK_PRESSURE_CUT_IN
and OFF at/above TANK_PRESSURE_CUT_OUT (e.g. on <=90, off >=120 psi). Run
the tank well above your highest tyre target so fills stay fast.

Diagnostics the tank sensor enables:
  - If the compressor runs longer than TANK_FILL_TIMEOUT_SECONDS without
    reaching cut-out, that's a failing pump or a major leak -> "POSSIBLE
    COMPRESSOR FAILURE".
  - If the tank sensor itself can't be read, the compressor is STOPPED as
    a fail-safe (don't run blind) and the same warning is raised.
  - A corner that can't reach target while the tank is healthy
    (TANK_SUPPLY_MIN_PSI+) is therefore a LOCAL problem -> puncture/valve,
    not the compressor. (This replaces the older "is the other axle ok?"
    guess with a real supply-pressure signal.)

ALWAYS keep a mechanical pressure switch wired as the hardware backstop,
set just above TANK_PRESSURE_CUT_OUT, plus a tank safety relief valve, in
case the sensor or software ever fails high.

Air plumbing checklist: tank rated for working pressure and vibration-
mounted; safety relief valve on the tank; check valve compressor->tank;
drain valve at the tank's low point (moisture); water trap/desiccant
downstream before the solenoid manifold; deflate vents to atmosphere.

============================================================
THREAD SAFETY
============================================================
All corners share a single ADS1115 instance. An ADC read is a non-atomic
write-config -> wait -> read-conversion sequence, so it is guarded by a
lock — without it, two corner threads reading different channels at once
would cross-wire their readings (and inflate/deflate the wrong tyre). The
lock is a no-op cost with one corner wired, but is required as soon as a
second corner is added.

============================================================
WIRING
============================================================
Air compressor (MOSFET power switch module):
    Pi GPIO4 -> module signal/IN pin
    The Pi only ENABLES the compressor; tank pressure (below) decides when it
    actually runs. For a high-current compressor, the MOSFET module should
    switch a properly-rated automotive relay/contactor that carries the pump
    on its own heavy-gauge, fused wiring near the battery — not the Pi rail.
    Keep a mechanical pressure switch in the circuit as the hardware backstop.

Corner pressure sensors -> DFRobot Gravity ADS1115 (I2C):
    FL sensor -> ADS1115 A0   FR sensor -> ADS1115 A1
    RL sensor -> ADS1115 A2   RR sensor -> ADS1115 A3
    ADS1115 SDA/SCL -> Pi SDA(GPIO2)/SCL(GPIO3), address 0x48 default

Tank pressure sensor (drives compressor control):
    Tank sensor -> ADS1115 channel TANK_SENSOR_CHANNEL at TANK_SENSOR_ADDR.
    Bench (only FL wired): share the 0x48 board on a free channel (A1 default).
    Production (all 4 corners on A0-A3): fit a SECOND ADS1115 with its ADDR pin
    tied to VDD (address 0x49) and set TANK_SENSOR_ADDR = 0x49, channel A0.

NOTE on sensor supply voltage: a 0.5-4.5V sensor read directly needs the
ADS1115 at ~5V, which puts 5V on the shared I2C bus into the Pi's 3.3V-only
GPIO2/3. Prefer running the ADS1115 (and OLED) at 3.3V; if the sensors are
RATIOMETRIC, power them at 3.3V too (range scales to ~0.33-2.97V, no divider —
adjust SENSOR_V_MIN / PSI_PER_VOLT) — otherwise add a divider on each signal or
an I2C level shifter. Confirm your sensor type before wiring.

Solenoids (each via its own relay/MOSFET driver -> 12V/24V supply):
    FL inflate -> GPIO17   FL deflate -> GPIO27   (<- wired now)
    FR inflate -> GPIO22   FR deflate -> GPIO23   (not yet wired)
    RL inflate -> GPIO24   RL deflate -> GPIO25   (not yet wired)
    RR inflate -> GPIO5    RR deflate -> GPIO6    (not yet wired)

    RELAY POLARITY: many cheap opto-isolated relay boards are ACTIVE-LOW
    (the load energises on a LOW signal). Set SOLENOID_ACTIVE_HIGH /
    COMPRESSOR_ACTIVE_HIGH in the script to match your board. Either way the
    de-energised (closed / compressor-off) state is what the code drives at
    startup — but BENCH-VERIFY that nothing energises at power-on before
    connecting air, since a wrong polarity setting would energise everything.

Buttons (momentary, wired to GPIO + GND, internal pull-up used):
    UP -> GPIO16   DOWN -> GPIO26   SELECT -> GPIO13

OLED display (SSD1309, I2C): shares the bus with the ADS1115 (different
    address — typically 0x3C or 0x3D). Assumed 128x64 resolution.

GPS (u-blox NEO-6M, UART/NMEA, 9600 baud default):
    Module VCC -> Pi 3.3V (pin 1)  — do NOT use 5V, chip is 3.3V-only
    Module GND -> Pi GND
    Module TX  -> Pi RXD (GPIO15 / physical pin 10)
    Module RX  -> not connected (only reading data, not configuring it)

============================================================
SETUP
============================================================
    sudo apt update
    sudo apt install -y python3-lgpio python3-smbus i2c-tools fonts-dejavu-core
    sudo raspi-config
        # Interface Options -> I2C -> Enable
        # Interface Options -> Serial Port ->
        #     "login shell over serial?" -> No
        #     "serial port hardware enabled?" -> Yes
        # reboot afterwards
    pip install gpiozero smbus2 luma.oled pillow pyserial pynmea2 --break-system-packages

    i2cdetect -y 1     # confirm ADS1115 (0x48) and SSD1309 (0x3C/0x3D)
    cat /dev/serial0    # should show $GPxxx NMEA sentences once GPS has a fix

Run as a systemd service (tyre-inflator.service, provided) so the
controller starts automatically at boot.
